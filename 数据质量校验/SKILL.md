# 数据质量检查 Skill (增强版)

## 核心原则（AI 必须遵守）

1. **安全第一**：绝对禁止在用户确认前执行任何数据库检查操作
2. **确认优先**：即使规则完全明确，也必须经过“展示 → 确认 → 执行”三步流程
3. **禁止跳过**：任何情况下不得直接调用 `dq_checker.py`，必须先获得用户明确确认
4. **一步一停**：每完成一个步骤后必须停止，等待用户下一次输入

## 强制执行流程（违反将导致不可信结果）

### 第一步：规则识别与引导（强制，最高优先级）
**校验对象**：仅校验用户输入的**原始问句文本**，不结合业务知识、不查数据库。

**校验规则**：
检查用户输入中是否包含具体的检查规则（表名 + 列名 + 约束条件）。
- **包含规则**  →  进入**第二步（整理规则并请求确认）**。
- **不包含任何规则**→ **拦截**，禁止继续往下执行。
- **拦截后立即 terminate，不做任何其他操作。** 不要继续推理，不要查询数据。

**拦截话术**（直接作为 terminate 的 answer，不附加任何文件）：

您还没有提供具体的数据质量检查规则。为了能为您提供精准的检查结果，请告诉我您关注的表和字段。

**您可以尝试以下两种方式：**

**方式 A：直接描述您的业务规则（推荐）**
您可以直接告诉我需要检查的内容，例如：
- **非空检查**：`检查 users 表的 email 字段不能为空`
- **唯一性检查**：`检查 users 表的 id 必须唯一`
- **值域检查**：`检查 orders 表的 status 只能是 ['待支付', '已支付', '已发货']`
- **范围检查**：`检查 products 表的 price 必须在 0 到 10000 之间`
- **格式检查**：`检查 users 表的 phone 必须符合手机号正则格式`
- **关联检查**：`检查 orders 表的 user_id 必须在 users 表的 id 中存在`

**方式 B：由 AI 自动分析推荐**
如果您不确定该检查什么，可以直接输入：**“分析数据库，自动生成规则”**。
我将扫描您的数据库表结构（Schema），根据字段类型和命名习惯，为您推荐一套候选的质量检查规则供您确认。

### 第二步：整理规则并请求用户确认（强制执行）
**本步骤为强制步骤，AI 不得以任何理由跳过。即使用户已提供完整规则，也必须在获得确认后才能执行**

当用户提供了规则后，AI 进行以下操作：
1. **解析规则**：将自然语言转换为内部格式（规则类型、表名、列名、参数）。
2. **生成表格**：输出一个清晰的 Markdown 表格，包含：【序号、规则类型、表名、列名、参数】。
3. **拦截**，禁止继续往下执行。立即 terminate，不做任何其他操作。** 不要继续推理，不要查询数据。

**拦截话术**（直接作为 terminate 的 answer，不附加任何文件）：
输出生成的表格，并询问用户是否确认，或者需要对规则修改

### 第三步：根据用户回复执行
- 如果用户回复 **“确认”** 或 **“是”** 或 **“确认全部”**：
  - 调用脚本执行检查：`python skills/data-quality-checker/scripts/dq_checker.py --config '<json>'`
  - **注意**：在构造 --config 参数的 JSON 时，**必须**包含完整的 database 配置对象（包含 type, host, port, user, password, database），即使环境变量中已存在相关信息。严禁仅传递 rules 而缺失 database 对象，否则会导致脚本崩溃。
  - 输出专业的数据质量检查报告（Markdown 格式，含摘要、检查规则说明、规则明细表、错误样例）。
  - **将上述 Markdown 内容写入本地文件**：
    - 使用 Agent 的文件写入工具（如 `write_file`）保存到 `./reports/dq_report_<YYYYMMDD_HHMMSS>.md`。
    - 写入前确保 `reports` 目录存在（若不存在，使用 `mkdir -p reports`）。
    - 写入成功后，解析出文件的绝对路径，在对话中输出以下信息（仅此一句）：
      > ✅ 检查完成！报告已保存至：`/absolute/path/to/reports/dq_report_xxx.md`，请直接下载该文件。
  - 报告输出后**停止**，等待用户下一次输入（流程回到第一步）。
- 如果用户回复**修改意见** $\rightarrow$ 根据意见更新规则 $\rightarrow$ 重新执行第二步。

## 特别说明：数据字典分析模式
如果用户请求“分析数据库”、“自动生成规则”或“根据表结构生成规则”：
1. AI 调用分析脚本：`python skills/data-quality-checker/scripts/dq_checker.py --config '<json>' --analyze`
2. 解析返回的 JSON，提取候选规则列表。
3. 将候选规则整理成表格（增加“生成原因”列）发送给用户。
4. 发送确认请求：“请回复‘确认全部’使用以上规则，或描述修改意见。”
5. **停止** $\rightarrow$ 后续流程与普通规则相同。

## 禁止行为（AI 必须遵守）
| 序号 | 禁止行为 | 后果 | 正确做法 |
| :---: | --- | --- | --- |
| 1 | 跳过第二步直接执行检查 | 检查结果不可信，可能误判 | 必须展示规则表格并等待“确认” |
| 2 | 未收到“确认”就调用脚本 | 可能执行错误的检查配置 | 检查上一条消息是否包含确认词 |
| 3 | 一次输出多步内容 | 违反“一步一停”原则 | 每步完成后调用停止逻辑 |
| 4 | 自行推断用户意图执行 | 可能误解规则含义 | 通过确认步骤对齐理解 |
| 5 | 修改用户原始规则 | 检查结果与用户预期不符 | 原样展示，让用户确认或修改 |

## 技术规格
### 支持的检查类型
| 规则类型 | 中文描述 | 必需参数 |
| :--- | :--- | :--- |
| `not_null` | 不能为空 | table, column |
| `unique` | 必须唯一 | table, column |
| `value_in` | 只能是 A,B,C | table, column, allowed_values |
| `range` | 在 min 到 max 之间 | table, column, min 和/或 max |
| `length` | 长度不超过 N | table, column, max_len |
| `regex` | 必须符合格式 | table, column, pattern |
| `foreign_key` | 必须在另一张表中存在 | table, column, ref_table, ref_column |
| `custom_sql` | 满足自定义 SQL 条件 | sql, expectation(必须为 Python 布尔表达式，如 'value is None' 或 'count == 0') |

### 数据库支持
- MySQL, PostgreSQL, SQLite, Oracle

### 所需环境变量
| 变量名 | 说明 | 必填 |
| :--- | :--- | :--- |
| `DQ_DB_TYPE` | mysql / postgresql / sqlite / oracle | 是 |
| `DQ_DB_HOST` | 数据库主机地址 | 条件必填 |
| `DQ_DB_PORT` | 数据库端口 | 否 |
| `DQ_DB_NAME` | 数据库名 | 是 |
| `DQ_DB_USER` | 数据库用户名 | 条件必填 |
| `DQ_DB_PASSWORD` | 数据库密码 | 条件必填 |

### 性能配置（可选）
用户可在规则中添加性能参数，AI 将其放入 JSON 的 `performance` 对象中。
- `max_rows_per_table`: 默认 10,000,000
- `sample_size`: 默认 1,000,000
- `max_workers`: 默认 4

## 注意事项
- 本技能只读不写，不会修改任何数据。
- 所有检查使用 SELECT 语句，不执行 INSERT、UPDATE、DELETE。
- 唯一性和外键检查默认全量执行以保证准确性。
- 支持使用只读数据库账户。
